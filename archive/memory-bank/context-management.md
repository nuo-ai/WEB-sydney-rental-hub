# 上下文管理与 Token 优化最佳实践

## 核心原则
我的运行效率和 Token 消耗是直接挂钩的。为了在整个项目生命周期中保持高性能和成本效益，我必须严格遵守以下上下文管理原则。我们的目标不是不消耗 Token，而是确保每一分 Token 都花在解决核心问题上，避免浪费在无关的上下文和重复劳动中。

## 工作流程

### 1. 分层上下文策略 (Tiered Context Strategy)
我必须将上下文视为一个三层系统，并有策略地使用它：
-   **长期记忆 (LTM) - 记忆库**:
    -   **目的**: 存放项目的核心事实、架构、关键决策和不常变更的技术背景。
    -   **协议**: 在每个新任务开始时，我**必须**首先阅读 `activeContext.md` 和 `progress.md`。然后，我将**按需、精准地**加载记忆库中的其他文件，只加载与当前任务**直接相关**的内容。严禁一次性加载所有记忆库文件。
-   **中期记忆 (MTM) - `activeContext.md`**:
    -   **目的**: 追踪当前工作周期的具体目标、实施计划、遇到的障碍和即时决策。这是我最常读写的文件。
    -   **协议**: 在每次执行工具**之前和之后**，我必须在 `<thinking>` 块中评估是否需要更新 `activeContext.md`，以准确反映我的工作状态和下一步计划。
-   **短期记忆 (STM) - 对话历史**:
    -   **目的**: 用于即时的、连续的交互。
    -   **协议**: 我必须认识到 STM 是昂贵且易失的。我不应依赖它来“记住”超过几轮对话的信息。任何重要的信息（如新的用户需求、关键决策）都必须**立即持久化**到 `activeContext.md` 中。

### 2. 任务驱动的上下文加载
-   在开始一个新任务（或一个主要步骤）时，我的第一个思考过程必须是：“完成这个任务所需的最小信息是什么？”
-   然后，我将制定一个计划，只读取包含这些最小必要信息的文件。例如，修复后端的一个 bug，我应该只加载 `backend/` 目录下的相关代码和 `techContext.md` 中与后端相关的部分，而不是整个项目的文件。

### 3. 主动进行上下文“垃圾回收”
-   **识别任务完成**: 当一个明确的子任务（如“数据库迁移”、“用户登录UI构建”）完成后，我应该主动向您建议进行一次“上下文整合”。
-   **执行整合与刷新**:
    1.  **总结成果**: 将已完成任务的关键产出、最终代码和重要决策，更新到相关的记忆库文件中。
    2.  **更新 `progress.md`**: 明确将该任务标记为完成。
    3.  **建议开启新任务**: 我应该建议您使用 `new_task` 工具开启一个新会话。这将清空当前的对话历史（STM），让我在一个干净、低成本的环境中开始下一个任务，只加载与新任务相关的上下文。

### 4. 隔离无关的“支线任务”
-   当遇到与核心任务无关的“支线任务”（如本地网络配置、工具安装失败）时，我必须有意识地将其隔离。
-   **协议**:
    1.  在 `activeContext.md` 中明确记录：“暂停主线任务，解决 [支线问题]”。
    2.  解决问题后，只将**最终的、可操作的结论**（例如，“数据库连接必须使用 pooler URL，已更新到 .env”）记录到相关的记忆库文件（如 `techContext.md`）或 `.clinerules` 文件中。
    3.  **严禁**将整个冗长的诊断过程（如多次 `ping` 或 `nslookup` 的尝试）保留在我的核心记忆中。问题解决后，应立即通过 `new_task` 等方式刷新上下文，回归主线任务。
